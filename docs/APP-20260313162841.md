## Part 1: AI Experience Questionnaire

### General Understanding

1. What does "AI-First SDLC" mean to you? How is it different from traditional development with AI tools added in? Give an example of how you've applied AI-First thinking in a project you've worked on.

    In this project I applied AI-First thinking by designing the Smart Backlog Assistant from the ground up as an agentic workflow - the agents aren't bolted onto a traditional SDLC tool, they are the SDLC process. The ActionPlanningAgent, RoutingAgent, and specialist agents (PM, Program Manager, Dev Engineer) map directly to the requirements → feature → task pipeline, with AI driving each transformation rather than assisting a human doing it manually.

2. Describe an AI solution you've built or contributed to. Walk through the key stages from idea to deployment. Where did AI help you the most in the process?

    I designed and built a multi-agent backlog generation system from scratch. Starting from a product specification, the system orchestrates an ActionPlanningAgent (which extracts the workflow steps), a RoutingAgent (which uses embedding-based cosine similarity to dispatch each step to the correct specialist agent), and three knowledge-augmented agents: a Product Manager agent for user stories, a Program Manager agent for feature grouping, and a Development Engineer agent for sprint-ready task cards. The system supports multiple LLM providers (OpenAI and Anthropic) through a unified AIClient abstraction. The most significant AI contribution was in the iterative prompt engineering cycle - multiple runs revealed failure modes (spec-detection gaps, dependency inversions, effort compression) that required precise prompt changes to fix, which is itself an AI-native debugging process.

    Other examples of projects I've worked on can be found on my GitHub public repositories.

3. How do you decide when to use AI versus traditional code for solving a problem? What factors do you consider? Give an example of when you chose traditional code over AI, and explain why.

    In the Smart Backlog Assistant, the routing logic is AI-driven (embedding similarity) rather than rule-based, because the decision space - mapping natural language step descriptions to agent responsibilities - is too semantically fluid for hard-coded rules. But the orchestration loop itself (run() method, context accumulation, file writing) is traditional code, because those are deterministic, structured operations where AI adds no value and would add unpredictability.

4. What are some advantages and limitations of using AI to automate tasks compared to writing traditional code? When have you seen each approach work well?

    AI automation is great for tackling complex, ever-changing tasks involving unstructured data, but it demands significant resources (like advanced systems and frameworks) and diverse data sources (from vector databases to traditional databases). In contrast, traditional coding shines for clear-cut, rule-based tasks like data processing, calculations, or adhering to strict regulations, offering transparency, efficiency, and precise control.

### Technical Depth

1. What is context in AI applications, and why does it matter? How do you manage context when building an AI solution? Describe a situation where you had to work with limited context (e.g., token limits) and how you handled it.

    Context in AI is crucial for maintaining coherence and relevance in interactions, managed through techniques like summarization, memory buffers, or external storage, and in token-limited scenarios, I've used chunking and prioritization to process long texts, like breaking thousand-words documents into hundred-words segments (chunks), with a (chunk size / 4) overlap, storing them in a vector database (Chroma DB), and creating tools for semantic search, allowing the model to reply the user's queries with ground-based answers (citations, references, etc.).

    RAGAs metrics were then used to verify the model's answers regarding faithfulness (how factually consistent a response is with the retrieved context), response relevance (how relevant a response is to the user input), Bleu score (measures the similarity between the response and the reference), and Rouge score (measures the overlap between the generated response and the reference text).

    In the Smart Backlog Assistant, I managed context across a three-step agentic pipeline by maintaining a context_sections list - a structured, labelled accumulation of each step's output (Product specification → User stories output → Features output). This ensured every agent received precisely the context it needed without unbounded growth or position-based ambiguity. Anchoring the spec as context_sections[0] guaranteed it was present for every agent regardless of step position.

2. How do you approach building memory or state management in an AI application? Give an example of when your AI solution needed to remember information across interactions.

    First, we need to identify context needs: determine what information needs to be retained across interactions, like user preferences, conversation history and task progress.

    Second, we need to choose the storage mechanism: short term memory, like in-memory session state for transient data, and long term memory, like databases, vector stores or file systems for long-term data.
    Third, we need to implement the retrieval mechanisms, like keywork matching, embeddings for semantic search, or state machines.

    Fourth, we need to measure and balance the memory depth/allocation. Example: retain only the user's last "n" messages in a conversation context, and expire their session after 30 minutes of inactivity.

    Fifth, we need to ensure scalability, optimizing memory storage and retrieval based on the business need.

    Finally, we must ensure that the edge cases have been accounted for, like session timeouts, data expiration or users' requests to clear the current context.

3. How do you track what your AI application is doing? Describe your approach to logging, monitoring, and debugging AI components. How do you know when something goes wrong?

    By combining logging (including key information like timestamps, input and output data, session/user ids and metadata - agent/workflow information), monitoring (such as real-time dashboards like Grafana or Prometheus), collecting models performance metrics (such as with RAGAs metrics - accuracy, precision, correctness, BLEU, ROUGE, and recall scores), and testing (writing unit tests for individual agents and tools, and integration tests for the whole workflow), etc.

    In practice, debugging the Smart Backlog Assistant required reading model outputs as diagnostic signals - the ActionPlanningAgent producing a single spec-request step was the signal that the spec wasn't reaching the planner; a feature-card output from a user-stories prompt was the signal for a routing mismatch; triple-duplicated content in the output document was the signal for missing deduplication in the orchestrator. Structured logging with step labels and truncated outputs at each routing call made these signals visible without overwhelming the console.

    A system designer/architect should consider agentic-AI automation as an experiment that needs to be under constant supervision and fine-tunning.

4. Describe a time when your AI solution produced an incorrect or unexpected result (hallucination). What did you do to identify the problem? What steps did you take to prevent it from happening again?

    There are several techniques one can use to avoid model hallucination, like prompt engineering - refining system prompts, tools descriptions, expected inputs and outputs; model tunning - like reducing the models temperature to achieve a more deterministic answer -; and RAG systems, which will provide the model with grounded information for its answers. I have yet to experience a model hallucinating in a real world application, but during my training, I've experienced this kind of behaviour and managed to reduce or eliminate it in open-end QA sessions by optimizing my system prompt, reducing the model's temperature and enforcing a token limit for the answers.

    In this project, GPT-5.4-mini consistently inverted the dependency between the SLA escalation task and the messaging connector task across multiple runs - it was not random hallucination but a systematic reasoning error caused by the spec mentioning Slack in two different sections. The fix required an explicit dependency direction rule in the agent knowledge string rather than a temperature reduction - set as 0.0 since the beginning -, which illustrates that some failure modes require structural prompt changes rather than parameter tuning.

5. How do you optimize your AI solution to use fewer tokens or API calls while still getting good results? What techniques have you used?

    During my training (Full stack FDE and nanodegree programs on Udacity) I managed to reduce the overall token utilization and API calls on my projects by applying the following:

    - Prompt engineering: crafting clear, concise and specific system prompts for all the agents. In some cases, agents were supposed to receive a verbose prompt (like the user's query), optimize it, and reply in a structured format (like JSON) only with the relevant information required for the next agent in the pipeline.
    - API call limits: in several examples I needed to implement a limit in how many times an agent (or a model) should retry calling an API; when certain threshold was reached, the agent was then forced to gracefully explain that it couldn't process the user's request at that time.
    - Model selection: during my training I had the chance to play with different models from providers like Open AI, Anthropic and Google. More expensive/capable models, like gpt-5.4 and sonnet-4.6, were used when reasoning capabilities were required, like agent orchestration/routing based on the user's input, and cheaper models, like gpt-3.5-turbo or gemini-2.5-flash, were used to perform the task at hand.
    - Asynchronous processing: in more than one occasion I saw myself calling agents and tools asynchronously in order to speed thing up. For instance, during a project for one of my Nanodegrees, I've implemented asynchronous calls in a chat bot to 1. load all the available embeddings collections in a vector database (Chroma DB); 2. perform a semantic search over that collection based on the users query; and 3. calculate the RAGAs metrics for the model's response. This implementation ensured that the chat-bot UI performed well, without being blocked by back-end processing.

    In the Smart Backlog Assistant, I addressed token efficiency through structured context labelling - passing only the relevant prior step output to each agent rather than the full conversation history. The spec was anchored once and reused across steps rather than re-injected from the user message each time. Agent knowledge strings were written to be as concise as possible while remaining unambiguous, since verbose knowledge increases every call's token cost proportionally to the number of workflow steps.

6. How do you handle retrieval of chunks in an AI application? Describe your approach to reranking retrieved results and stitching them into a coherent response. What trade-offs do you consider?

    I retrieve relevant chunks using semantic search (cosine similarity), re-rank them based on context and query relevance (top-n result available for the model's context), and stitch them into a coherent response by identifying overlapping themes and ensuring logical flow, balancing accuracy, conciseness, and readability. It doesn't mean, though, that the model can't hallucinate even with the ground-based provided context, and that's why prompt engineering techniques, model selection, logging and observability is paramount in any gen AI/agentic AI implementation.

3. Describe an agent pattern or AI workflow you've implemented. How did you structure it? What made it work well (or what challenges did you face)?

    The most important architectural decision was the separation between the ActionPlanningAgent (which determines what to do) and the RoutingAgent (which determines who does it). This separation meant that adding a new agent type required only registering a new description in the routing table - the planner and orchestrator needed no changes. It also meant that prompt failures in one agent didn't cascade into others, making debugging significantly easier.

4. Explain the difference between "Human-in-the-Loop" and "Human-on-the-Loop" in your own words. Give an example of when you'd use each approach in software development.

    In "Human-in-the-Loop", an human is actively involved in the decision-making process; in other words, the system cannot complete the task at hand without an human input/intervention. Example: an agentic system for content moderation can flag inappropriate content, but maybe isn't always accurate; it then flags this content for human moderators, who then decide if the content should be allowed or blocked.

    On the other hand, in "Human-on-the-Loop", an human "oversees" the system operation but isn't directly involved in every decision: the system operates autonomously, and the human steps in only when necessary - to correct errors, handle exceptions, or improve the system's performance. Example: an automated customer support chat-bot, which handles most of the users' inquiries autonomously, but escalates the case to an human agent depending of the complexity or priority of the task at hand.

    The Smart Backlog Assistant operates in Human-on-the-Loop mode by design - the pipeline runs autonomously and produces a complete backlog document, but a human (BA, PM or tech lead) reviews the output before it enters a sprint planning session. The `--spec` input and the evaluation cycle (multiple runs reviewed and corrected through prompt changes) is itself a Human-on-the-Loop pattern: the human doesn't intervene in individual agent decisions, but reviews aggregate outputs and adjusts the system's behaviour through prompt engineering.
